# Email Split — Choice B (Baseline Proof)

## Core rule (approved)

- **Strategic email** = **byte-identical** to current output: same `buildPurchaseConfirmationEmail` / `buildReferrerCommissionEmail`, same `applyGlobalEmailBanner`, same `client.messages.send` payload shape.
- **Trust email** = **new, additive only** — minimal transactional; **does not replace** the strategic send.

## Proof artifacts

Generated by `../render-email-baseline-proof.cjs`:

```bash
cd deepquill
# Clean baseline (no contest banner in subject/body)
set EMAIL_BANNER=0   # Windows CMD
node scripts/render-email-baseline-proof.cjs

# Same as typical production when EMAIL_CONTEST_BANNER=1
set EMAIL_BANNER=1
node scripts/render-email-baseline-proof.cjs
```

Outputs:

| Folder | Meaning |
|--------|---------|
| `no-banner/` | `EMAIL_CONTEST_BANNER` effectively off (`''`); subject **without** `[PUBLIC BETA TEST]` prefix |
| `with-banner/` | `EMAIL_CONTEST_BANNER=1`; subject/body get banner treatment |

Each folder contains:

- `purchase.subject.txt`, `purchase.text.txt`, `purchase.html`
- `referral.subject.txt`, `referral.text.txt`, `referral.html`
- `LINKS.json` — URLs + mailto extracted from HTML+text

## Strategic vs baseline (post-change)

Under Choice B, **proposed strategic email** must match the **same** pipeline as today. Proof:

1. Run this script **before** and **after** adding `sendTrustEmail` (strategic path unchanged).
2. `diff` `no-banner/purchase.*` (and `referral.*`) — **must be empty** for strategic output.
3. Repeat for `with-banner/` if production uses banner env.

## Snapshot test plan (implementation phase)

1. Commit golden files from `email-split-proof/no-banner/` (and optionally `with-banner/`) as Jest snapshots or checked-in fixtures.
2. In CI, call builders + `applyGlobalEmailBanner` with **fixed fixtures** (same as this script) and `assert.equal` to golden strings.
3. **Fail** on any byte difference in subject, text, or html for the strategic path.

## Rollback

- Feature flag or single code path: if trust email disabled, only strategic send runs — **identical to today**.
- No DB schema required for Choice B (trust send can log to existing patterns only if you add optional ledger rows later).

## Implementation (live)

**Flag:** `EMAIL_SPLIT_TRUST_EMAIL=1` in the **deepquill** / webhook environment.

**From address (trust only):** `no-reply@theagnesprotocol.com` by default. Override with `MAILCHIMP_TRUST_FROM_EMAIL` if needed. Strategic emails still use `MAILCHIMP_FROM_EMAIL`.

- When **unset** or not `1`: behavior matches the previous single-email flow (strategic only).
- When **`1`**: Stripe webhook sends a **short trust** email first, then the **unchanged** strategic email (`build*` → `applyGlobalEmailBanner` → Mailchimp). Trust failure is logged; strategic send still runs.

**Code:** `src/lib/trustTransactionalEmail.cjs` (builders only); `api/stripe-webhook.cjs` (send order: trust → strategic).

**Verify strategic parity (CI / pre-deploy):**

```bash
cd deepquill
npm run email:verify-strategic
```

**Regenerate goldens** (after intentional template changes only):

```bash

```

**Do not** edit `purchaseEmail.cjs`, `referrerCommissionEmail.cjs`, or `emailBanner.cjs` for strategic parity unless explicitly approved and goldens updated.
